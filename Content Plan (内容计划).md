## 选题逻辑

围绕一个母题：**「用 AI IDE 把能力编排成可复用的自动化工作流，解决真实场景问题。」**

筛选一个选题是否成立，必须同时满足：

1. **正向可成**：AI 真能把这件事做成（不做以黑功能为目的的选题）。
2. **操作层可复现**：有清晰的步骤 / Prompt / 配置 / 命令，观众照着能跑通。
3. **判断层有料**：能如实交代边界与验收——什么前提下成立、哪一步要人接管、怎么验收。
4. **可沉淀**：产出能复用（角色 / 工作流 / 模板 / 组件），不是一次性 demo。
5. **反噱头**：不夸大常规环境坑，不以代码行数当卖点（脚本 Schema 的 `anti_hype_forbidden` 会机器拦截标题噱头）。

> 头号选题来源就是**本仓库自己的生产线**：把「如何用 AI IDE 搭一条角色 × 工作流 × 状态机驱动的视频流水线」拆成系列内容（Dogfooding，以行证言）。

## 标准化生产流水线（13 阶段状态机）

每期都走同一条流水线。工作流真相源在 `shared/workflows/<slug>.md`（IDE 无关，经 `scripts/sync_workflows.py` 物化到各 IDE 目录，`/<slug>` 调用），产物按阶段落 `content-library/epNN-slug/`：

```
01 选题 → 02 策划 → 03 B站视听 → 04 脚本(SSOT) → 05 B轨录屏⏸ → 06 TTS
   → 07 组装 → 08 字幕 → 09 BGM⏸ → 10 封面⏸ → 11 质检⏸ → 12 分发⏸ → 13 归档⏸
```

- **看板门禁（L0.5 单核校验）**：阶段置 `approved` = **Schema 校验通过 + 人工确认**。
  - 机器校验 `scripts/pipeline_lint.py` 比对产物尾部 ` ```json ` 块与 `shared/schemas/*.json`（01/02/03/04），并做跨阶段防漂移（`04 脚本`为 SSOT、上游状态一致、场景数一致、标题不触犯噱头黑名单）。
  - 判断层评审（CHAI 第二核）当前**已挂起**，文件保留待恢复。
- **断点续跑**：重启时读看板，定位第一个 `-`/`draft` 阶段续跑。
- **角色驱动**：每个阶段由 [`shared/roles/`](./shared/roles/) 的对应角色作为 system prompt 执行（选题分析师→内容策划师→视听策划师→文案撰稿人→录制导演→语音/字幕/混音/封面/质检/分发）。

## 两轨制作（对应 TAD-01 混合架构）

技术教程必须 100% 真实，因此按类型分两条制作轨道，最大化 Remotion 自动化、收窄人工断点：

| 轨道 | 适用 | 制作方式 | Remotion 角色 | 人工核心 |
|------|------|---------|--------------|---------|
| **A 轨** | 概念/解说/总结型 | 脚本 → Remotion 场景组件 → 直接渲染成片 | 全自动渲染 | 几乎零录屏（口播可录可 TTS） |
| **B 轨** | 实操演示型 | 真实 IDE 录屏 + 真人/TTS 口播 → 套编译器组件 | 片头/片尾/转场/字幕/数据卡/多尺寸 | 真实录屏 + 高光标注 |

- B 轨原料（Raw MP4）作为 `cuts` 的 `OffthreadVideo` 嵌入；WhisperX 字级转录 + 智能混音由 Agent 自动完成。
- 15–30s 的聚焦 UI 演示可用 `terminal_scene` / `screenshot_scene` **合成录屏**替代真实录屏。
- **红线**：凡 Remotion 无法生成、需真实操作的画面，一律 `[B 轨占位：请提供 xxx.png/mp4]` 显式标出；绝不虚构 AIGC 界面/数字人对口型/AI 绘技术图。

## 内容支柱：Video-as-Code / 流程即代码

一根可跨期复用的主线（也是频道差异化护城河）：

- **代码即视频**：视频不是画面堆砌，而是「参数输入 + 场景渲染」——脚本里直接备注「此处调用 `remotion-composer` 的 `@ComparisonCard` / `@TerminalScene`，传入 JSON 参数」。
- **流程即代码**：整条生产线被做成 `角色（system prompt）× 工作流（user prompt 模板）× frontmatter+看板（状态机）`，因此天然**易被 Agent 接管 / API 化**。
- **以行证言（Dogfooding）**：口播可直接指着屏幕说「这张高亮代码卡片和字级弹跳字幕是 OpenMontage 后台自动生成的」——用内容本身的自动化证明硬核可信度。

## 自动化成熟度路线

- **L0.5 单核校验（当前）**：Schema 契约 + 人工确认；看板手动更新；判断层评审挂起。
- **L1 半自动（规划）**：选题/策划自动提炼实操并自动跑 Schema 校验。
- **L2 全自动（规划）**：全链路多角色 Agent 自动校验、评审、流转（对应 `OpenMontage/pipeline_defs/*.yaml` 的可执行管道）。

## 分发矩阵

成片统一渲染 1080p；Remotion 可一次产出 16:9 与 9:16 多尺寸，竖版直接喂短视频切片。

| 平台 | 形式 | 语言 | 优先级 | 核心作用 |
|------|------|------|--------|---------|
| B 站 | 长视频（3–8 分钟） | 中文 | ⭐⭐⭐⭐⭐ | 主阵地，完整承载操作层 + 判断层 |
| GitHub | 配套代码 / Prompt / 模板 | 中英 | ⭐⭐⭐⭐ | 信任背书，精准开发者受众（本仓库即配套） |
| 小红书 | 图文卡片 + 短视频切片 | 中文 | ⭐⭐⭐⭐ | 引流，操作层轻量化呈现 |
| 知乎 | 深度文章 | 中文 | ⭐⭐⭐ | SEO 长尾，理性决策人群 |
| 抖音 | 短视频切片（高光） | 中文 | ⭐⭐ | 纯引流，强钩子导流 B 站 |
| YouTube / Dev.to | 长视频 / 技术文章 | 英文 | ⭐⭐ | 海外开发者，扩大基本盘（后续） |

**节奏**：聚焦 B 站 + 小红书 + GitHub 三角跑通内容模型，其余渠道待模型验证后开。

### 短视频切片策略

短视频是专门的「钩子产品」，不是长视频截取。生产流程：

```
B站成片（9:16 竖版已由 Remotion 渲染）
  → 人工标注高光时间点 → FFmpeg 自动切片
  → Agent 生成平台专用标题 + hook 文案 → 分发抖音/小红书
```

| 切片类型 | 时长 | 设计 |
|---------|------|------|
| 反预期开场 | 15–30s | 先抛反直觉结论 |
| 高光操作 | 30–60s | 最有视觉冲击的自动化过程 |
| 数据冲击 | 10–15s | 对比数字（手动 vs 自动） |

> 抖音只抛钩子、不做完整判断层；判断层引流到 B 站完整版。
