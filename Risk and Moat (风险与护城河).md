## AGI抗风险分析

| 层级 | AGI后的命运 | 我们做哪层 | 说明 |
|------|-----------|----------|------|
| 操作层（怎么点） | ❌ 会被替代 | 只是载体 | 界面变了内容就废了 |
| 判断层（该不该用/怎么验收） | ✅ 长期有效 | **核心** | 用户永远需要决策参考 |
| 场景定义层（把需求转化为AI任务） | ✅ 长期有效 | **核心** | 需求拆解是人的能力 |
| 工具评测层（哪个工具适合什么场景） | ✅ 工具越多越需要 | **核心** | 工具越多选择越难 |

**潜在风险**：如果AI IDE在1-2年内做到真正"零门槛"（用户无需任何判断就能得到完美结果），判断层价值会下降。

**对冲**：场景工作流地图本身是动态资产——工具越强/越多，地图反而需要更频繁更新，构成持续内容引擎。

## 核心判断记录

> 记录关键决策和背后逻辑，避免未来重复纠结。

1. **选Cursor为主线**：中国用户最多（50万+），可控性强，适合教程录制，步骤清晰可见
2. **Windsurf为副线**：中文内容空白大，做对比评测覆盖更多用户
3. **IDE设计>底层模型**：即使两个工具调用同一个底层模型，效果也不同——IDE的设计决定70%的体验（上下文注入方式、任务规划、错误恢复机制）
4. **不做纯编程内容**：市场已饱和，非编程场景才是差异化点
5. **角色体系跳过阶段1，直接用 Windsurf Workflows 验证**（2026.05.28）：
   - 原计划：手动复制 .md → .cursor/rules/ → 工作流编排（三阶段）
   - 改为：手动复制 .md → **Windsurf Workflows**（直接跳到阶段3）
   - 理由：用户已在用 Windsurf；Workflows 门槛低（纯 Markdown）；端到端跑通才能验证"AI IDE 撑起内容生产线"这个频道核心命题
   - 关键约束：拆成3个独立 Workflow（topic-research / script-draft / distribute-rewrite），单点跑通后再串联，避免一次做完整流程的累积失败率
6. **`.cursor/rules/` 与 Windsurf 不互通**：Cursor 私有格式 `.mdc`，Windsurf 不读；Windsurf 用 `.windsurf/rules/` 和 `.windsurf/workflows/`。`shared/roles/*.md` 作为跨工具源文件，两边各自适配。**这一差异本身是第5期对比评测的重要素材**。
7. **双 IDE 同步落地的结构映射**（2026.06.01）：把 Cursor 体系同步到 Windsurf 时，验证了两者的对应关系——
   - Cursor 的 `alwaysApply: true` 规则（`project-context`、`role-system`）→ Windsurf 的 `.windsurf/rules/` + `trigger: always_on`
   - Cursor 的工作流 `.mdc`（`alwaysApply: false`，手动触发）→ Windsurf 的 `.windsurf/workflows/*.md`（`/命令` 触发）
   - **共同的单一事实源是 `shared/roles/*.md`**，两套 IDE 配置都只做"指向 + 适配"，不复制角色正文，避免双份维护
   - **踩坑**：同步时发现旧的 `.windsurf/workflows/01` 引用了 `strategist.md`（无中文后缀）已失效——重命名角色文件后跨配置引用没同步更新，这是"分散规则 vs 可编排工作流"的真实代价，可作为第9期角色库公开期的反面案例素材
8. **内容策划工作流拆分为"故事策划(02)"与"B站视听编排(03)"**（2026.06.XX）：
   - **背景**：`02-content-planning-bilibili` 同时承载了"人类创意层"（故事结构、Demo 设计、判断层）与"技术执行层"（Remotion 组件映射、Props 接口、动画 Cue），导致 AI 在创意阶段被迫输出大量技术细节，分散了对核心故事质量的注意力。
   - **决策**：将 02 重命名为 `02-content-planning`，只保留故事结构、Demo 路径、A/B 轨配比、判断层；新增 `03-video-planning-bilibili` 与【视听策划师】角色，专门负责 Remotion 技术编排。
   - **好处**：
     - 人类在 02 阶段只需关注"讲什么故事、暴露什么坑"，不受技术组件限制。
     - AI 在 03 阶段全自动完成组件映射与 Props 生成，充分发挥其在结构化技术规格上的优势。
     - 为未来多平台（抖音竖屏、YouTube 横屏）扩展留下清晰的"同一故事、不同技术编排"的复用空间。
   - **关键约束**：04 脚本撰写的上游从 02 改为 03（或同时依赖 02+03），确保口播文案与最终 Remotion 组件参数完全对齐。
9. **独立 `/video` 渲染层与 OpenMontage (Python) 分离决策**（2026.06.09）：
   - **决策**：拒绝将所有 Remotion 组件合并到 OpenMontage 的 `remotion-composer` 内，坚持使用独立顶层 `/video` 目录（React 19 + Remotion 4.0.470），与 OpenMontage 内部组件库（React 18 + Remotion 4.0.441）物理隔离。
   - **理由**：
     - **依赖安全**：防止 React 18 vs 19 的 peer dependencies 冲突，确保教程代码能自由升级。
     - **开发 DX**：在顶层 `/video` 运行 `npm run studio` 独立调试和预览 `TimelineScene`/`TableScene` 更快、更纯净，不受 OpenMontage 庞大通用组件的干扰。
     - **开源解耦**：本频道的配套 Remotion 模板库是开源主打资源，解耦后更有利于独立打包并在 GitHub 归档。
   - **接口桥梁**：OpenMontage 仅作为素材采集（录屏、语音、字幕时间戳）和决策流水线，其输出的 JSON 资产通过简单脚本或手动配置注入顶层 `/video` 的 `data.ts`。
