## 2026.06.02 | 第一篇：内容流水线重构与 API 自动化可行性思考

### 1. 今日核心重构决策

今天我们对项目的目录结构进行了一次重大重构，建立了**“思路脑暴（Ideas）”**与**“生产成品（Content Library）”**彻底分离的机制。

#### 为什么做这个重构？（Pain Points）
在极简的 Flat-file (纯文件) 架构下，AI 很容易把未定稿的脑暴素材与定稿混在一起。如果只有一个目录，看板 `PIPELINE.md` 的状态控制会极其混乱。

#### 解决方案：两库分离
- `ideas/`：脑暴库。任何不成熟的想法、零碎的信息都堆在这里（如 `backlog.md`）。格式随意，可自由涂抹，AI 无需对此负格式责任。
- `content-library/`：成品库。必须严格按阶段落盘（01选题 -> 02脚本 -> 03组装 -> 04分发）。每个文件顶部带 YAML frontmatter 来标识状态（`stage` / `status: approved`）。
- **看板控制（PIPELINE.md）**：成为进度的唯一真相（Single Source of Truth），只追踪成品库中通过人工审核（L0 阶段全人审）的内容。

### 2. 内容方法论的微调（定位守护）

#### 痛点与思考
原定位包含“AI 不能做什么”，但作为教程频道，如果每期都是“AI 做不好的吐槽”，信息价值偏低，且观众看完无法落地。
因此，我们将核心定位修改为：**正向验证成功的应用场景**。

#### 关键妥协与护城河保留
但是，为了防止频道滑向没有差异化的“AI 吹捧/软广号”，我们**重新定义了“判断层”**：
- **旧定义**：判断层 = 吐槽 AI 做不好的点。
- **新定义**：判断层 = **边界与验收 / 避坑指南**。
即便是 100% 成功的场景，也必须交代：它在什么前提下成立？哪一步极易掉链子/需要人接管？验收标准是什么？
> **金句素材**：*“即便是滤镜下的 demo 跑得再顺，没有严格边界限制的智能，在复杂业务中也是脱缰的野马。”*

### 4. 关于后续“API 自动化”的技术选型思考（核心素材）

当项目跑通后，如果我们要将这套流水线从 IDE 交互改成后台 API 自动化运行，技术难度有多大？
**结论是：难度极低，因为目前的架构就是为此设计的。**

#### 为什么易于 API 化？
1. **Prompt 模板解耦**：
   我们的 `shared/roles/`（角色 SOP）天然就是 API 的 `system_prompt`。
   我们的 `.windsurf/workflows/`（工作流步骤）天然就是 `user_prompt` 的模板。
2. **状态机文件化**：
   通过文件的 Frontmatter 标记阶段（`stage: 02-script` / `status: approved`），一个简单的 Python 脚本（配合 `python-frontmatter` 库）就可以当作状态机调度器（Orchestrator），读到 `approved` 就自动发请求调用 LLM 跑下一阶段，并自动改写状态。
3. **Remotion 的命令行友好性**：
   Remotion 视频框架天生支持 CLI（`npx remotion render`），可以通过 `subprocess` 甚至云端的 AWS Lambda 实现 100% 自动化一键渲染。

#### API 化的真正难点
1. **多轮交互（Feedback Loop）**：
   在 IDE 里，我们可以直接跟 AI 聊天微调脚本。变成 API 后，需要引入一个“反馈输入口”（比如一个简单的网页 Web UI）让人输入修改意见，API 接收后自动更新对应的 markdown 文件。
2. **多模态物理限制**：
   A 轨（概念动画）可全自动，但 B 轨（实操录屏）需要人上传音频/录屏。API 流程必须设计“挂起等待”机制。

### 5. 今日架构演进：02-script 平台化拆分

#### 决策
为了与 04-distribute（分发）保持一致，也为了防范单个 markdown 文件过大、结构混乱，我们决定将 `02-script.md` 升级为一个目录 `02-script/`：
- `02-script/video.md`（视频主脚本）
- `02-script/zhihu.md`（知乎，后续）
- `02-script/xiaohongshu.md`（小红书，后续）

这进一步强化了**“文件即数据（File-as-data）”**的微服务化思想。
